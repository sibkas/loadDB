# Тестирование производительности: WordPress и БД (миллион комментариев)

Проект содержит сценарии нагрузочного тестирования WordPress с использованием MySQL-процедур для генерации сверхбольших объемов данных и определения точки отказа.

## 📂 Структура проекта
* `wp/` — конфигурация Docker и файлы ядра WordPress.
* `wp_db_test.jmx` — сценарий JMeter (Readers + Writers).
* `scripts.sql` — SQL-скрипты: процедуры `dorepeat_v1`, `dorepeat_v2` и триггер для обновления счетчика.
* `screenshots/` — скриншоты результатов, разделенные по сценариям:
    * `v1/`, `v2/` — сравнение методов генерации данных.
    * `load20sec/` — поиск точки отказа с нуля.
    * `load1M20sec/` — поиск точки отказа при старте с 1 млн записей.
* `Dashboard_Report.../` — детальные HTML-отчеты JMeter.


## 📈 Результаты тестирования

### 1. Сравнение методов наполнения (v1 vs v2)
Проведен анализ производительности двух версий SQL-процедур.
* **Результат:** Процедура `v1` показала большую стабильность/скорость при массовой вставке.
* [Скриншот v1](screenshots/v1/Response%20Times%20Over%20Time.png)
* [Скриншот v2](screenshots/v2/Response%20Times%20Over%20Time.png)

### 2. Поиск точки отказа (Старт с 1 комментария)
Постепенное наполнение базы до превышения SLA (20 секунд).
* **Результат:** Точка отказа достигнута при накоплении **~1.95 млн записей**.
* [Скриншот Response Time](screenshots/load20sec/Response%20Times%20Over%20Time.png)
* [Скриншот счетчика (20s)](screenshots/load20sec/20sec.png)
* [Скриншот счетчика коментариев](screenshots/load20sec/total_count.png)

### 3. Тестирование на 1 000 000 записей
Замер производительности при уже нагруженной БД.
* **Результат:** При наличии 1 млн записей базовое время отклика составило **~9 сек**. Деградация до 20 сек наступила при достижении общего объема ~3.00 млн записей.
* [Скриншот 1 млн](screenshots/load1M20sec/Response%20Times%20Over%20Time.png)
* [Скриншот счетчика (20s)](screenshots/load1M20sec/20sec.png)
* [Скриншот счетчика коментариев](screenshots/load1M20sec/total_count.png)
